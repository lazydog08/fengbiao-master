# AGENTS.md - 封标大师

## 发布 / 部署默认规则

除非用户明确说“不要发布”“不要部署”“先别上线”或提出等价限制，完成一个任务并完成必要验证后，默认要把当前前后端状态同步到线上网页。

这条规则是用户给出的长期授权：满足验证条件后，Codex 可以直接发布线上网页，不需要每次另行询问。但未完成、未验证、测试失败、构建失败或核心流程仍有阻塞的改动不算“完成”，不得自动发布。

### 默认流程

- 公开站点由 `gh-pages` 分支的 GitHub Pages 提供，不是直接由 `main` 提供；把代码推到 `main` 不等于网页已经更新。
- 标准发布命令是 `./scripts/publish_github_pages.sh`。它会导出前端快照、拷贝/压缩网页封面、构建 `apps/web`，并推送 `apps/web/dist` 到 `gh-pages`。
- 仅前端改动也要运行标准发布命令，让线上网页反映改动。
- 涉及后端、抓取逻辑、数据库、分类、封面缓存、导出快照或其他会影响公开站点数据的改动时，优先使用 `./scripts/publish_github_pages.sh --sync`，让数据刷新、快照导出和前端发布一起完成。
- 任意发布都会从当前本机数据库重新导出公开快照；发布前必须确认本机数据库状态适合公开上线，不能把临时、实验或脏数据误发布。
- `--sync` 会执行真实公开数据抓取；即使抓取失败，脚本也可能继续导出并发布当前本机快照，最后用非零退出码标记失败。遇到这种情况必须明确说明“抓取/同步失败，但可能已发布旧快照或部分数据”。

### 发布后校验

- 发布后必须核对线上 Pages URL 是否已经加载新资源或新数据；不要只确认 `main` 已 push。
- 如果 GitHub Pages 仍在 `building`，等待它变为 `built` 后再做线上校验。
- 线上校验优先检查真实页面行为、当前 JS/CSS 资源文件名、关键 UI 文案或快照数据是否已经更新。
- 发布脚本只更新 `gh-pages`，不会自动提交或推送源码到 `main`；源码或配置有变更时，还要单独把相关源码提交并推送到 `main`，避免线上站点和仓库源码漂移。

### 提交边界

- 工作区存在无关脏文件时，只暂存、提交、推送本次任务相关文件。
- 不要为了发布网页而把无关后端、数据、文档或实验文件混进提交。
- `gh-pages` 发布产物由发布脚本管理；不要手工改 `apps/web/dist` 后直接复制上线。
- 发布脚本使用非交互 Git 凭据；如果 push 鉴权失败，要把它归为发布/凭据问题，不要误报为代码实现失败。

### 失败处理

- 如果实现完成但发布失败，要把“实现完成”和“发布失败”分开汇报。
- 如果发布脚本失败、GitHub Pages 构建失败、线上校验仍是旧版本，必须明确说明失败环节和是否需要用户介入。

### 例外

- 用户明确表示不发布时，跳过发布流程，只完成代码改动，并在结论里说明“本次按要求未发布到线上网页”。
